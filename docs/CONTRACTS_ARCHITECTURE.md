# YieldVault-RWA: Contracts Architecture Overview

**Target Network:** Stellar Soroban  
**Language:** Rust (compiled to WebAssembly)  
**soroban-sdk Version:** 22.0.0  
**Last Updated:** May 29, 2026

---

## 1. Project Overview

YieldVault-RWA is a decentralized vault protocol built on Stellar's Soroban smart contracts. It enables users to deposit USDC and earn yield generated by tokenized real-world assets (RWAs) such as Korean sovereign debt instruments. The protocol implements:

- **ERC-4626-style vault** with fractional share minting (`yvUSDC`)
- **Multi-strategy support** via pluggable strategy connectors (BENJI, Korean Debt)
- **DAO governance** for strategy selection via weighted voting
- **RWA shipment tracking** for physical asset provenance
- **Protocol fees** with treasury accumulation
- **Large-withdrawal timelocks** for risk management
- **Per-user deposit caps** and minimum deposit thresholds
- **Oracle price validation** (infrastructure ready, not yet integrated)

## 3. Module Responsibilities

### 3.1 YieldVault (Main Contract)

**File:** `contracts/vault/src/lib.rs`  
**Purpose:** Core vault logic implementing ERC-4626 pattern with RWA-specific features

#### Public Interface

**Initialization & Admin:**
- `initialize(admin, token)` вҖ” One-time initialization with admin and underlying token
- `propose_admin(new_admin)` вҖ” Two-step admin transfer (propose)
- `accept_admin()` вҖ” Accept pending admin role
- `upgrade(new_wasm_hash)` вҖ” WASM code upgrade (admin-only)

**Deposit & Withdrawal:**
- `deposit(user, amount) -> Result<i128, VaultError>` вҖ” Mint shares for deposited tokens
- `withdraw(user, shares) -> Result<i128, VaultError>` вҖ” Burn shares, return assets (may create timelock)
- `execute_withdrawal(user) -> Result<i128, VaultError>` вҖ” Complete pending large withdrawal after timelock

**Strategy Management:**
- `set_strategy(strategy)` вҖ” Set active strategy (must be whitelisted)
- `whitelist_strategy(strategy, approved)` вҖ” Add/remove strategy from whitelist
- `is_strategy_whitelisted(strategy) -> bool` вҖ” Check if strategy is whitelisted
- `strategy() -> Option<Address>` вҖ” Get active strategy address
- `invest(amount)` вҖ” Move idle funds to strategy
- `divest(amount)` вҖ” Recall funds from strategy

**Yield & Accrual:**
- `accrue_yield(amount)` вҖ” Admin-initiated yield accrual (deducts protocol fee)
- `report_benji_yield(strategy, amount)` вҖ” BENJI strategy callback to report yield
- `accrue_korean_debt_yield() -> i128` вҖ” Harvest yield from Korean debt strategy

**Governance (DAO):**
- `set_dao_threshold(threshold)` вҖ” Set voting threshold for proposals
- `create_strategy_proposal(proposer, strategy) -> u32` вҖ” Create strategy proposal
- `vote_on_proposal(voter, proposal_id, support, weight)` вҖ” Vote on proposal
- `execute_strategy_proposal(proposal_id)` вҖ” Execute approved proposal (sets BenjiStrategy)

**RWA Shipment Tracking:**
- `add_shipment(shipment_id, status)` вҖ” Add new shipment (admin-only)
- `update_shipment_status(shipment_id, new_status)` вҖ” Update shipment status (admin-only)
- `shipment_ids_by_status(status, cursor, page_size) -> ShipmentPage` вҖ” Paginated shipment query

**Protocol Fees (Goal 1):**
- `set_fee_bps(new_bps)` вҖ” Set protocol fee in basis points (0вҖ“10000)
- `fee_bps() -> i128` вҖ” Get current fee rate
- `set_treasury(treasury)` вҖ” Set treasury address
- `treasury() -> Option<Address>` вҖ” Get treasury address
- `treasury_balance() -> i128` вҖ” Get accumulated fee balance

**Large-Withdrawal Timelock (Goal 2):**
- `set_large_withdrawal_threshold(threshold)` вҖ” Set threshold for 24-hour timelock
- `large_withdrawal_threshold() -> i128` вҖ” Get current threshold

**Minimum Deposit (Goal 3):**
- `set_min_deposit(new_min)` вҖ” Set minimum deposit amount
- `min_deposit() -> i128` вҖ” Get current minimum

**Oracle Configuration (Planned):**
- `set_price_oracle(oracle)` вҖ” Set oracle contract address
- `price_oracle() -> Option<Address>` вҖ” Get oracle address
- `set_oracle_enabled(enabled)` вҖ” Enable/disable oracle validation
- `is_oracle_enabled() -> bool` вҖ” Check if oracle is enabled
- `set_oracle_heartbeat(seconds)` вҖ” Set oracle staleness threshold
- `oracle_heartbeat() -> u64` вҖ” Get oracle heartbeat

**Pause/Unpause:**
- `pause()` вҖ” Pause vault (blocks deposits/withdrawals)
- `unpause()` вҖ” Resume vault
- `is_paused() -> bool` вҖ” Check pause status

**Query Functions:**
- `token() -> Address` вҖ” Get underlying token address
- `total_shares() -> i128` вҖ” Get total vault shares outstanding
- `total_assets() -> i128` вҖ” Get total assets (idle + strategy value)
- `balance(user) -> i128` вҖ” Get user's share balance
- `calculate_shares(assets) -> i128` вҖ” Calculate shares for asset amount
- `calculate_assets(shares) -> i128` вҖ” Calculate assets for share amount
- `per_user_cap() -> i128` вҖ” Get per-user deposit cap
- `user_deposit(user) -> i128` вҖ” Get user's total deposit amount
- `benji_strategy() -> Address` вҖ” Get configured BENJI strategy
- `korean_strategy() -> Address` вҖ” Get configured Korean debt strategy

#### Error Enum (VaultError)

```rust
pub enum VaultError {
    AlreadyInitialized = 1,      // initialize() called twice
    InsufficientShares = 2,       // withdraw() with more shares than user has
    InvalidAmount = 3,            // deposit/withdraw with amount <= 0
    ContractPaused = 4,           // deposit/withdraw while paused
    ExceedsUserCap = 5,           // deposit exceeds per-user cap
    MinDepositNotMet = 6,         // deposit below minimum
    TimelockNotExpired = 7,       // execute_withdrawal() before timelock
    NoPendingWithdrawal = 8,      // execute_withdrawal() with no pending
}
```

**Call Flow:**
1. **User deposits USDC** вҶ’ YieldVault calls `TokenSAC.transfer()` to receive tokens
2. **Admin invests** вҶ’ YieldVault calls `Strategy.deposit()` to move funds
3. **Strategy yields** вҶ’ Strategy calls `YieldVault.report_benji_yield()` to report yield
4. **Admin accrues yield** вҶ’ YieldVault calls `KoreanDebtStrategy.harvest_yield()` to get yield
5. **User withdraws** вҶ’ YieldVault calls `TokenSAC.transfer()` to return tokens (may divest from strategy)

## 5. Data Flow: User Deposit Journey

For full request-to-confirmation sequence diagrams covering both deposit and withdrawal paths (including the 24-hour timelock flow), see **[Deposit & Withdrawal Lifecycle](./DEPOSIT_WITHDRAWAL_LIFECYCLE.md)**.


```
User Deposit Flow:
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ

1. User calls vault.deposit(user, 100 USDC)
   в”ңв”Җ Check: amount > 0, not paused, meets min_deposit
   в”ңв”Җ Check: user_deposit + 100 <= per_user_cap
   в”ңв”Җ Calculate: shares_to_mint = 100 * total_shares / total_assets
   в”Ӯ            (or 100 if first deposit)
   в”ңв”Җ Effect: Update state
   в”Ӯ   в”ңв”Җ total_assets += 100
   в”Ӯ   в”ңв”Җ total_shares += shares_to_mint
   в”Ӯ   в”ңв”Җ user_balance[user] += shares_to_mint
   в”Ӯ   в””в”Җ user_deposit[user] += 100
   в”ңв”Җ Interaction: TokenSAC.transfer(user, vault, 100)
   в””в”Җ Event: emit ("deposit", (100, shares_to_mint))

2. User calls vault.withdraw(user, shares_to_mint)
   в”ңв”Җ Check: shares > 0, user_balance[user] >= shares, not paused
   в”ңв”Җ Calculate: assets_to_return = shares * total_assets / total_shares
   в”ңв”Җ Check: if assets_to_return > large_withdrawal_threshold
   в”Ӯ   в””в”Җ Create pending withdrawal with 24-hour timelock
   в”Ӯ   в””в”Җ Event: emit ("pndwdraw", user, (shares, unlock_ts))
   в”Ӯ   в””в”Җ Return 0 (user must call execute_withdrawal later)
   в”ңв”Җ Otherwise (normal withdrawal):
   в”Ӯ   в”ңв”Җ Effect: Update state
   в”Ӯ   в”Ӯ   в”ңв”Җ total_assets -= assets_to_return
   в”Ӯ   в”Ӯ   в”ңв”Җ total_shares -= shares
   в”Ӯ   в”Ӯ   в”ңв”Җ user_balance[user] -= shares
   в”Ӯ   в”Ӯ   в””в”Җ user_deposit[user] -= assets_to_return (capped at 0)
   в”Ӯ   в”ңв”Җ Interaction: TokenSAC.transfer(vault, user, assets_to_return)
   в”Ӯ   в””в”Җ Event: emit ("withdraw", user, (assets_to_return, shares))
   в””в”Җ Return: assets_to_return

3. (If timelocked) User calls vault.execute_withdrawal(user)
   в”ңв”Җ Check: pending withdrawal exists, unlock_timestamp <= now
   в”ңв”Җ Effect: Remove pending withdrawal, update state as above
   в””в”Җ Interaction: TokenSAC.transfer(vault, user, assets_to_return)
```

## 6. Storage Architecture

### Instance Storage (Persistent)

All vault state is stored in **instance storage** (persistent across contract calls):

- **Admin & Initialization:** `Admin`, `Initialized`, `PendingAdmin`
- **Core Vault:** `TokenAsset`, `TotalShares`, `TotalAssets`, `State`
- **Strategy:** `Strategy`, `StrategyWhitelist(Address)`, `BenjiStrategy`, `KoreanDebtStrategy`
- **User Balances:** `ShareBalance(Address)`, `UserDeposit(Address)`
- **Governance:** `DaoThreshold`, `ProposalNonce`, `Proposal(u32)`, `Vote(u32, Address)`
- **Shipments:** `ShipmentByStatus(ShipmentStatus)`, `ShipmentStatusOf(u64)`
- **Fees & Treasury:** `FeeBps`, `Treasury`, `TreasuryBalance`
- **Withdrawals:** `LargeWithdrawalThreshold`, `PendingWithdrawal(Address)`
- **Deposits:** `MinDeposit`, `PerUserCap`
- **Oracle:** `PriceOracle`, `OracleEnabled`, `OracleHeartbeat`
- **Pause:** `IsPaused` (stored in `State` struct)

### TTL & Extend Strategy

Soroban instance storage has a default TTL of ~6 months. The vault does **not** explicitly extend TTL вҖ” it relies on Soroban's automatic extension on contract invocation. For production, consider:

- Periodic "heartbeat" transactions to extend TTL
- Monitoring TTL expiration via ledger state

### Shared Storage Patterns

- **User-keyed storage:** `ShareBalance(Address)`, `UserDeposit(Address)`, `Vote(u32, Address)`, `PendingWithdrawal(Address)` вҖ” allows efficient per-user queries
- **Status-keyed storage:** `ShipmentByStatus(ShipmentStatus)` вҖ” enables filtering by status
- **Proposal-keyed storage:** `Proposal(u32)` вҖ” indexed by proposal ID

## 7. Security Model

### Authorization Boundaries

| Function | Required Auth | Notes |
|----------|---------------|-------|
| `initialize` | None | One-time, sets admin |
| `upgrade` | Admin | WASM upgrade |
| `propose_admin` / `accept_admin` | Admin / PendingAdmin | Two-step transfer |
| `set_strategy` | Admin | Must be whitelisted |
| `whitelist_strategy` | Admin | |
| `pause` / `unpause` | Admin | |
| `configure_korean_strategy` | Admin | |
| `accrue_korean_debt_yield` | Admin | |
| `set_dao_threshold` | Admin | |
| `add_shipment` / `update_shipment_status` | Admin | |
| `accrue_yield` | Admin | |
| `invest` / `divest` | Admin | |
| `set_fee_bps` / `set_treasury` | Admin | |
| `set_large_withdrawal_threshold` | Admin | |
| `set_min_deposit` | Admin | |
| `set_price_oracle` / `set_oracle_enabled` / `set_oracle_heartbeat` | Admin | |
| `create_strategy_proposal` | Proposer (signed) | Any user can propose |
| `vote_on_proposal` | Voter (signed) | Any user can vote |
| `execute_strategy_proposal` | Public | Anyone can execute |
| `report_benji_yield` | Configured BenjiStrategy | Only registered strategy |
| `deposit` | User (signed) | User must sign |
| `withdraw` / `execute_withdrawal` | User (signed) | User must sign |
| Query functions | Public | No auth required |

### Reentrancy Protections

**Soroban's Atomic Model:**
- Contract calls are atomic within a transaction
- State changes are committed atomically
- No recursive calls can occur during execution
- Each contract invocation gets its own execution frame

**CEI Pattern (Checks-Effects-Interactions):**
All state-changing functions follow CEI:
1. **Checks:** Validate inputs, auth, preconditions
2. **Effects:** Update contract state
3. **Interactions:** Make external calls (token transfers, strategy calls)

Example from `deposit()`:
```
Checks:
  - amount > 0
  - not paused
  - meets min_deposit
  - user_deposit + amount <= cap

Effects:
  - Update total_assets, total_shares, user_balance, user_deposit

Interactions:
  - TokenSAC.transfer(user, vault, amount)
```

### Admin/Owner Controls

- **Two-step admin transfer:** `propose_admin()` + `accept_admin()` prevents accidental loss
- **Pause mechanism:** Admin can pause vault to stop deposits/withdrawals during emergencies
- **Strategy whitelist:** Admin controls which strategies can be set
- **Governance threshold:** Admin sets DAO voting threshold

### Input Validation

- **Amount validation:** All amounts checked for > 0 (or >= 0 for min_deposit)
- **Overflow protection:** All arithmetic uses `checked_*` methods
- **Share rounding:** Deposits that would mint 0 shares are rejected (prevents silent loss)
- **Threshold validation:** Large-withdrawal threshold must be > 0

## 8. Developer Guide

### Adding a New Contract Module

1. **Create module file:** `contracts/vault/src/my_module.rs`
2. **Declare in lib.rs:**
   ```rust
   pub mod my_module;  // or #[cfg(test)] mod my_module; for test-only
   ```
3. **Define public interface:**
   ```rust
   pub fn my_function(env: Env, param: Type) -> Result<ReturnType, MyError>
   ```
4. **Add storage keys if needed:**
   ```rust
   // In DataKey enum in lib.rs
   MyModuleKey(Address),
   ```
5. **Document with RustDoc:**
   ```rust
   /// Brief description of what this does.
   ///
   /// ### Parameters
   /// * `param` - What this parameter does
   ///
   /// ### Returns
   /// What is returned
   ///
   /// ### Errors
   /// When errors occur
   pub fn my_function(...)
   ```

### Running Tests

```bash
# Run all tests
cargo test

# Run specific test
cargo test test_deposit_works

# Run with output
cargo test -- --nocapture

# Run fuzz tests (10,000 iterations)
cargo test fuzz_deposit_withdraw_symmetry_no_fee

```

## 9. Events & Webhooks

### Event Emissions

YieldVault emits cryptographically-signed events for all critical operations. These events are published to the Stellar blockchain and can be consumed by off-chain indexers, backend services, and user notification systems.

#### Event Catalog

| Event | Emitted By | When | Topics | Data |
|-------|-----------|------|--------|------|
| `deposit` | `deposit()` | User deposits USDC | contract_id | (amount, shares_minted) |
| `pndwdraw` | `withdraw()` | Large withdrawal initiated (timelocked) | contract_id, user | (shares, unlock_timestamp) |
| `withdraw` | `withdraw()` / `execute_withdrawal()` | Withdrawal completes | contract_id, user | (assets_returned, shares_burned) |
| `feechg` | `set_fee_bps()` | Protocol fee updated | contract_id | (old_bps, new_bps) |
| `mindepchg` | `set_min_deposit()` | Minimum deposit updated | contract_id | (old_min, new_min) |

#### Event Details

**`deposit` Event**
- **Emitted:** When user successfully deposits USDC
- **Data:** `(amount: i128, shares_minted: i128)`
- **Use Case:** Track user deposits, update analytics, trigger notifications

**`pndwdraw` Event**
- **Emitted:** When withdrawal exceeds `large_withdrawal_threshold` (24-hour timelock)
- **Data:** `(shares: i128, unlock_timestamp: u64)`
- **Topics:** Includes user address for filtering
- **Use Case:** Notify user of pending withdrawal, track timelock expiry

**`withdraw` Event**
- **Emitted:** When withdrawal completes (either immediately or after timelock)
- **Data:** `(assets_returned: i128, shares_burned: i128)`
- **Topics:** Includes user address for filtering
- **Use Case:** Track user withdrawals, update balances, trigger notifications

**`feechg` Event**
- **Emitted:** When admin updates protocol fee
- **Data:** `(old_bps: i128, new_bps: i128)`
- **Use Case:** Update fee configuration, alert on significant changes

**`mindepchg` Event**
- **Emitted:** When admin updates minimum deposit threshold
- **Data:** `(old_min: i128, new_min: i128)`
- **Use Case:** Update deposit validation, alert on threshold changes

## 10. Known Limitations & Future Work

### Current Limitations

1. **Oracle not integrated:** `oracle.rs` module is ready but not wired into vault logic
   - Functions `set_price_oracle()`, `is_oracle_enabled()`, `oracle_heartbeat()` exist but are not used
   - Strategy value validation against oracle is planned but not implemented

2. **Single active strategy:** Only one strategy can be active at a time
   - Future: Support multiple strategies with allocation percentages

3. **No strategy performance fees:** Strategies don't take a cut of yield
   - Future: Add strategy fee configuration

4. **Shipment tracking is basic:** No integration with external RWA provenance systems
   - Future: Connect to Stellar Attestation Service or similar

5. **No emergency withdrawal:** Users cannot withdraw during pause
   - Future: Add emergency withdrawal with reduced share price

### Planned Features

- [ ] Oracle price validation integration
- [ ] Multi-strategy allocation
- [ ] Strategy performance fees
- [ ] Emergency withdrawal mechanism
- [ ] Yield distribution events
- [ ] Cross-chain bridge support

### Key Invariants Tested

1. **Share price consistency:** `total_assets / total_shares` ratio never changes unexpectedly
2. **Full exit zeroes state:** When all users withdraw all shares, state reaches 0
3. **Sum of balances:** Total shares = sum of all user balances
4. **Yield never changes shares:** Yield accrual only increases total_assets
5. **No value extraction:** Deposit вҶ’ withdraw never returns more than deposited
6. **Overflow safety:** All arithmetic operations are safe
